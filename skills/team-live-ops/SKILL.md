## 团队构成
- **live-ops-designer** — 赛季结构、活动节奏、保留机制、战斗通行证
- **经济设计师** — 实时经济平衡、商店轮换、货币定价、怜悯计时器
- **分析工程师** — 成功指标、A/B 测试设计、事件跟踪、仪表板规格
- **社区经理** - 面向玩家的公告、活动描述、季节性消息
- **叙事导演** — 季节性叙事主题、故事情节、世界事件框架
- **作家** — 活动描述、奖励物品名称、季节风味文字、公告文案

## 如何委派

使用 Task 工具将每个团队成员生成为子代理：
- `subagent_type: live-ops-designer` — Season/event 结构和保持机制
- `subagent_type: economy-designer` — 实时经济平衡和奖励定价
- `subagent_type: analytics-engineer` — 成功指标、A/B 测试、事件检测
- `subagent_type: community-manager` — 面向玩家的通信和消息传递
- `subagent_type: narrative-director` — 季节性主题和叙事框架
- `subagent_type: writer` — 所有面向玩家的文本：事件描述、项目名称、副本

始终在每个代理的提示中提供完整的背景信息（游戏概念路径、现有赛季文档、道德政策路径、当前经济状况）。在管道允许的情况下并行启动独立代理（第 3 阶段和第 4 阶段可以同时运行）。

## Pipeline

### 第 1 阶段：Season/Event 范围界定
委托给 **live-ops-designer**：
- 定义季节或活动：类型（季节性、限时活动、挑战）、持续时间、主题方向
- 概述内容列表：新增内容（模式、项目、挑战、故事节奏）
- 定义保留挂钩：本赛季是什么让球员回归 daily/weekly
- 确定资源预算：需要创建多少新内容与重用多少新内容
- 输出：季节简介，包括范围、内容列表和保留机制概述

### 第二阶段：叙事主题
委托给**叙事导演**：
- Read 第一阶段的赛季简介
- 设计季节叙事主题：这个事件如何与游戏世界联系起来？
- 定义玩家在活动期间将发现的中心故事情节
- 确定本赛季哪些现有的故事线索可以推进
- 输出：叙事框架文档（主题、故事挂钩、背景故事联系）

### 第 3 阶段：经济设计（如果主题明确，则与第 2 阶段并行）
委托给**经济设计师**：
- Read `design/live-ops/economy-rules.md` 的赛季简介和现有经济规则
- 设计奖励途径：免费等级升级、高级等级价值主张
- 规划季节性经济：季节性货币、商店轮换、定价
- 为任何随机元素定义怜悯计时器机制和厄运保护
- 验证高级赛道中没有付费获胜项目
- 输出：包含奖励表、定价和货币流动的经济设计文档

### 第 4 阶段：分析和成功指标（与第 3 阶段并行）
委托给**分析工程师**：
- Read 赛季简介
- 定义成功指标：参与率目标、保留提升目标、战斗通行证完成率
- 设计在赛季期间运行的任何 A/B 测试（例如，不同的奖励节奏）
- 指定本季内容所需的新遥测事件
- 输出：包含成功标准和仪器要求的分析计划

### 第五阶段：内容写作（并行）
并行委托：
- **叙事导演**（如果需要）：Write 本季的任何游戏内叙事文本（过场脚本、NPC 对话、世界事件描述）
- **作者**：Write 所有面向玩家的文本 - 事件名称、奖励物品描述、挑战目标文本、季节风味文本
- 两人都应该阅读第二阶段的叙事框架文档

### 第 6 阶段：玩家沟通计划
委托给**社区经理**：
- Read 本季简介、经济设计和叙事框架
- 起草赛季发布公告（基调、主要亮点、特定平台版本）
- 规划沟通节奏：发布前预告片、发布日帖子、季中提醒、最后一周 FOMO 推送
- 第一天补丁说明的已知问题部分占位符草稿
- 输出：通信日历以及每个接触点的草稿副本

### 第 7 阶段：审核和签署
收集所有阶段的产出并提出综合的季节计划：
- 季节简介（第一阶段）
- 叙事框架（第二阶段）
- 经济设计和奖励表（第3阶段）
- 分析计划和成功指标（第 4 阶段）
- 书面内容清单（第 5 阶段）
- 沟通日历（第 6 阶段）

向用户呈现摘要：
- **内容范围**：正在创建的内容
- **经济健康检查**：奖励轨道是否感觉公平且非掠夺性？
- **分析准备情况**：成功标准是否已定义和检测？
- **道德审查**：根据 `design/live-ops/ethics-policy.md` 检查第 3 阶段经济设计
  - 如果该文件不存在：标记“道德审查已跳过：未找到 `design/live-ops/ethics-policy.md`。经济设计未根据道德政策进行审查。建议在生产开始前创建一个。”将此标志包含在季节设计输出文档中。添加到后续步骤：创建 `design/live-ops/ethics-policy.md`。
  - 如果文件存在并且发现违规行为：标记“道德标志：第 3 阶段经济设计中的 [element] 违反了 [policy rule]。在解决此问题之前，将阻止批准。”不要发布完整的判决或编写输出文档。使用 `AskUserQuestion` 和以下选项：修改经济设计/根据记录的理由覆盖/取消。如果用户选择修改：重新生成经济设计器以生成更正的设计，然后返回到第 7 阶段审核。
- **悬而未决的问题**：在生产开始之前仍需要做出决定

在委托给制作团队之前，请用户批准季节计划。仅在用户批准且不存在未解决的道德违规行为后才发布完整裁决。如果道德违规行为未得到解决，则以判决结束：**被阻止**。

## 输出文件

所有文档保存到 `design/live-ops/`：
- `seasons/S[N]_[name].md` — 季节设计文档（来自第 1-3 阶段）
- `seasons/S[N]_[name]_analytics.md` — 分析计划（来自第 4 阶段）
- `seasons/S[N]_[name]_comms.md` — 通信日历（从第 6 阶段开始）

## 错误恢复协议

如果任何生成的代理（通过 Task）返回 BLOCKED、错误或无法完成：

1. **立即浮出水面**：在继续依赖阶段之前向用户报告“[AgentName]：BLOCKED — [reason]”
2. **评估依赖关系**：检查后续阶段是否需要被阻止的代理的输出。如果是，则在没有用户输入的情况下不要继续越过该依赖点。
3. **通过 AskUserQuestion 提供选项**，可选择：
   - 跳过此代理并注意最终报告中的差距
   - 以更窄的范围重试
   - 停在这里并首先解决阻止程序
4. **始终生成部分报告** — 输出已完成的内容。切勿因为一名代理阻塞而放弃工作。

如果 BLOCKED 状态无法解决，则以判决结束： **BLOCKED** 而不是 COMPLETE。

## 文件 Write 协议

委托给通过 Task 生成的子代理。每个子代理强制执行
“我可以写信给 [path] 吗？”协议。该编排器不直接写入文件。
