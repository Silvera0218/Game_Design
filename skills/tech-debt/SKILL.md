## 第 1 阶段：Parse 子命令

从参数确定模式：

- `scan` — 扫描代码库以获取技术债务指标
- `add` — 手动添加新的技术债务条目
- `prioritize` — 重新确定现有债务登记册的优先级
- `report` — 生成当前债务状态的摘要报告

如果没有提供子命令，则输出用法并停止。结论：**失败** — 缺少必需的子命令。

---

## 阶段 2A：扫描模式

在代码库中搜索债务指标：

- `TODO`评论（统计和分类）
- `FIXME` 评论（这些是伪装成债务的错误）
- `HACK` 评论（需要适当解决方案的解决方法）
- `@deprecated` 标记
- 重复的代码块（多个文件中的类似模式）
- 文件超过500行（潜在神物）
- 函数超过 50 行（潜在复杂性）

对每个发现进行分类：

- **架构债务**：错误的抽象、缺失的模式、耦合问题
- **代码质量债务**：重复、复杂性、命名、缺失类型
- **测试债务**：缺少测试、不稳定的测试、未经测试的边缘情况
- **文档债务**：缺少文档、过时的文档、未记录的 API
- **依赖债务**：过时的软件包、不推荐使用的 API、版本冲突
- **性能债务**：已知的慢速路径、未优化的查询、内存问题

将调查结果呈现给用户。

问：“我可以将这些发现写入 `docs/tech-debt-register.md` 吗？”

如果是，请更新寄存器（附加新条目，不要覆盖现有条目）。结论：**完成** — 扫描结果写入寄存器。

如果没有，就停在这里。结论：**被阻止**——用户拒绝写入。

---

## 阶段 2B：添加模式

提示：描述、类别、受影响的文件、估计的修复工作量、未修复时的影响。

向用户呈现新条目。

问：“我可以将此条目附加到 `docs/tech-debt-register.md` 吗？”

如果是，请附加该条目。结论：**完成** — 条目已添加到寄存器中。

如果没有，就停在这里。结论：**被阻止**——用户拒绝写入。

---

## 阶段 2C：优先模式

Read 债务登记册位于 `docs/tech-debt-register.md`。

每个项目的评分依据：`(impact_if_unfixed × frequency_of_encounter) / fix_effort`

按优先级分数重新排序注册表，并建议下一个冲刺中包含哪些项目。

将重新确定优先级的寄存器呈现给用户。

问：“我可以将重新确定优先级的寄存器写回到`docs/tech-debt-register.md`吗？”

如果是，则写入更新的文件。结论：**完成** — 寄存器重新确定优先级并保存。

如果没有，就停在这里。结论：**被阻止**——用户拒绝写入。

---

## 第 2D 阶段：报告模式

Read 债务登记册。生成汇总统计数据：

- 按类别划分的总项目数
- 预计总修复工作量
- 自上次报告以来添加的项目与已解决的项目
- 趋势方向（增长/稳定/收缩）

标记登记册中超过 3 个冲刺的任何项目。

将报告输出给用户。此模式是只读的 - 不写入任何文件。结论：**完成** — 生成债务报告。

---

## 第三阶段：后续步骤

- 运行 `/sprint-plan` 将高优先级债务项目安排到下一个冲刺中。
- 在每个冲刺开始时运行 `/tech-debt report` 以跟踪一段时间内的债务趋势。

### 债务登记册格式

```markdown
## Technical Debt Register
Last updated: [Date]
Total items: [N] | Estimated total effort: [T-shirt sizes summed]

| ID | Category | Description | Files | Effort | Impact | Priority | Added | Sprint |
|----|----------|-------------|-------|--------|--------|----------|-------|--------|
| TD-001 | [Cat] | [Description] | [files] | [S/M/L/XL] | [Low/Med/High/Critical] | [Score] | [Date] | [Sprint to fix or "Backlog"] |
```

### Rules
- 科技债务本质上并不是坏事——它只是一种工具。登记册追踪有意识的决定。
- 每项债务条目都必须解释其被接受的原因（截止日期、原型、缺失信息）
- “扫描”应该在每个冲刺中至少运行一次以捕获新债务
- 超过 3 个冲刺但没有采取行动的项目应该被修复或有意识地接受并记录原因
