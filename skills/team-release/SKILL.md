1. Read `production/session-state/active.md` 和 `production/milestones/` 中的最新文件（如果存在）来推断目标版本。
2. 如果找到版本：报告“未提供版本参数 - 从里程碑数据推断 [version]。继续。”然后用 `AskUserQuestion` 确认：“释放 [version]。这是正确的吗？”

## 团队构成
- **release-manager** — 发布分支、版本控制、变更日志、部署
- **qa-lead** — 测试签核、回归套件、发布质量门
- **devops-engineer** — 构建管道、工件、部署自动化
- **安全工程师** - 预发布安全审核（如果游戏具有 online/multiplayer 功能或玩家数据则调用）
- **analytics-engineer** — 验证遥测事件正确触发并且仪表板处于活动状态
- **社区经理** - 补丁说明、发布公告、面向玩家的消息
- **生产者** — Go/no-go 决策、利益相关者沟通、调度

## 如何委派

使用 Task 工具将每个团队成员生成为子代理：
- `subagent_type: release-manager` — 发布分支、版本控制、变更日志、部署
- `subagent_type: qa-lead` — 测试签核、回归套件、发布质量门
- `subagent_type: devops-engineer` — 构建管道、工件、部署自动化
- `subagent_type: security-engineer` — online/multiplayer/data 功能的安全审核
- `subagent_type: analytics-engineer` — 遥测事件验证和仪表板准备情况
- `subagent_type: community-manager` — 补丁说明和发布通讯
- `subagent_type: producer` — Go/no-go 决策，利益相关者沟通
- `subagent_type: network-programmer` — 网络代码稳定性签核（如果游戏有多人游戏则调用）

始终在每个代理的提示中提供完整的上下文（版本号、里程碑状态、已知问题）。在管道允许的情况下并行启动独立代理（例如，第 3 阶段代理可以同时运行）。

## Pipeline

### 第一阶段：发布计划
委托给**生产者**：
- 确认满足所有里程碑验收标准
- 确定此版本中推迟的任何范围项目
- 设定目标发布日期并与团队沟通
- 输出：发布授权并确认范围

### 第 2 阶段：发布候选版本
委托给**发布经理**：
- 从商定的提交中删除发布分支
- 在所有相关文件中添加版本号
- 使用 `/release-checklist` 生成发布清单
- 冻结分支 - 没有功能更改，仅修复错误
- 输出：发布分支名称和清单

### 第 3 阶段：质量门（并行）
并行委托：
- **qa-lead**：执行完整的回归测试套件。测试所有关键路径。验证没有 S1/S2 错误。签署质量。
- **devops-engineer**：为所有目标平台构建发布工件。验证构建是否干净且可重现。在 CI 中运行自动化测试。
- **安全工程师** *（如果游戏具有在线功能、多人游戏或玩家数据）*：进行发布前安全审核。审查身份验证、反作弊、数据隐私合规性。签署安全态势。
- **网络程序员** *（如果游戏有多人游戏）*：签署网络代码稳定性。验证负载下的延迟补偿、重新连接处理和带宽使用情况。

### 第 4 阶段：本地化、性能和分析
委托（如果资源可用，可以与第 3 阶段并行运行）：
- 验证所有字符串均已翻译（委托给 **localization-lead** 如果可用）
- 针对目标运行性能基准（委托给 **性能分析师** 如果可用）
- **分析工程师**：验证所有遥测事件在发布版本上正确触发。确认仪表板正在接收数据。检查关键渠道（入职、进展、货币化（如果适用））是否已检测。
- 输出：本地化、性能和分析签核

### Phase 5: Go/No-Go
委托给**生产者**：
- 收集来自以下人员的签核：qa-lead、release-manager、devops-engineer、安全工程师（如果在第 3 阶段产生）、网络程序员（如果在第 3 阶段产生）和技术总监
- 评估任何未解决的问题 - 它们是否会阻碍或可以交付？
- 拨打 go/no-go 电话
- 输出：发布带有理由的决定

**如果生产者声明不予通过：**
- 立即做出决定：“制作人：不可以——[理由，例如，在第 3 阶段发现的 S1 错误]。”
- 使用 `AskUserQuestion` 和选项：
  - 修复阻塞程序并重新运行受影响的阶段
  - 将发布推迟到稍后日期
  - 用书面理由推翻 NO-GO（用户必须提供书面理由）
- **完全跳过第 6 阶段** — 不标记、部署到暂存、部署到生产或生成社区管理器。
- 生成一份部分报告，总结阶段 1-5 以及跳过的内容（阶段 6）及其原因。
- 结论：**被阻止** — 版本未部署。

### 第 6 阶段：部署（如果进行）
委托给 **release-manager** + **devops-engineer**：
- 在版本控制中标记发布
- 使用 `/changelog` 生成变更日志
- 部署到阶段进行最终冒烟测试
- 部署到生产环境
- 发布后48小时监控

委托给 **community-manager**（与部署并行）：
- 使用 `/patch-notes [version]` 完成补丁说明
- 准备发布公告（商店页面更新、社交媒体、社区帖子）
- 如果有任何 S3+ 问题发布，请草拟已知问题
- 输出：所有面向玩家的发布通信，准备在部署确认时发布

### 第 7 阶段：发布后
- **release-manager**：生成发布报告（已发布的内容、推迟的内容、指标）
- **生产者**：更新里程碑跟踪，与利益相关者沟通
- **qa-lead**：监控传入的错误报告以进行回归
- **社区经理**：发布所有面向玩家的通信，监控社区情绪
- **分析工程师**：确认实时仪表板正常；如果缺少任何关键事件，则发出警报
- 如果出现问题，安排发布后回顾

## 错误恢复协议

如果任何生成的代理（通过 Task）返回 BLOCKED、错误或无法完成：

1. **立即浮出水面**：在继续依赖阶段之前向用户报告“[AgentName]：BLOCKED — [reason]”
2. **评估依赖关系**：检查后续阶段是否需要被阻止的代理的输出。如果是，则在没有用户输入的情况下不要继续越过该依赖点。
3. **通过 AskUserQuestion 提供选项**，可选择：
   - 跳过此代理并注意最终报告中的差距
   - 以更窄的范围重试
   - 停在这里并首先解决阻止程序
4. **始终生成部分报告** — 输出已完成的内容。切勿因为一名代理阻塞而放弃工作。

常见的拦截器：
- 输入文件丢失（未找到故事，GDD 不存在）→ 重定向到创建它的技能
- ADR 状态为建议 → 不实施；首先运行 `/architecture-decision`
- 范围太大 → 通过 `/create-stories` 分成两个故事
- ADR 和故事之间的指令冲突 → 冲突浮出水面，不要猜测

## 文件 Write 协议

所有文件写入（发布清单、变更日志、补丁说明、部署脚本）
委托给子代理和子技能。每个都强制执行“我可以写入 [path] 吗？”
协议。该编排器不直接写入文件。

结论：**完成** — 版本已执行并部署。
