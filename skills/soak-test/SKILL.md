# 浸泡测试

浸泡测试（也称为耐力测试）是长时间的游戏运行
具有特定的观察目标。与烟雾检查不同（广泛的关键路径，
~10 分钟）或单功能游戏测试（~30 分钟），浸泡测试运行 **30
分钟到几个小时** 浮出水面：

- **内存泄漏** - 仅在场景转换后才会出现的逐渐堆增长
- **性能漂移** — 帧时间退化随着时间的推移而恶化
- **状态累积错误** — 仅在 N 次重复后出现的问题
  机械师（库存满、分数溢出、AI状态损坏）
- **有趣的疲劳** - 在第一次训练中感觉良好但会不断成长的机制
  重复性过长的游戏
- **内容耗尽**——玩家耗尽新奇内容的程度

**此技能生成观察协议和分析工具 -
人类进行实际的演奏。**

**输出：** `production/qa/soak-test-[date]-[duration].md`

**何时运行：**
- 抛光阶段 — `/gate-check release` 之前
- 修复内存或稳定性问题后（回归浸泡）
- 当长时间比赛尚未被正式追踪时

---

## 1. 解析参数

**持续时间**（默认值：`1h`）：
- `30m` — 短时间浸泡；适合测试单个机制或场景
- `1h` — 标准浸泡；涵盖最常见的泄漏类别
- `2h` — 延长浸泡时间；建议首次完全抛光浸泡
- `4h` — 深度浸泡；具有长会话设计的游戏（角色扮演游戏、模拟人生）所需

**焦点**（默认：`all`）：
- `memory` — 关注堆大小、对象计数、泄漏模式
- `stability` — 重点关注crash/freeze/hang检测
- `balance` — 专注于乐趣疲劳、内容耗尽、困难感知
- `all` — 以上所有

## 4. 生成浸泡测试协议

### 记忆/稳定性观察项目（如果焦点=记忆或全部）

特定于发动机的监控指南：

**Godot 4:**
- 打开调试器→监视器选项卡；跟踪 `Memory → Static Memory` 和
  `Object Count → Objects` 跨检查点
- 记录：静态内存 (KB)、对象计数、孤立节点计数
- 警报阈值：前 15 分钟后从 T+0 开始内存增长 > 20%
  （预计负载会有所增长；持续增长表明存在泄漏）
- 注意：`Performance.get_monitor(Performance.MEMORY_STATIC)` 返回字节
  in Godot 4.6

**Unity:**
- 打开内存分析器（窗口→分析→内存分析器）
- 记录：总预留内存 (MB)、GC 分配 (MB)、每个检查点的对象计数
- 警报阈值：GC 分配在 3 个以上检查点上单调增长

**Unreal 引擎：**
- 在每个检查点使用 `stat memory` 控制台命令
- 记录：已用物理内存（MB）、可用物理内存
- 警报阈值：物理内存使用增长超过完全浸泡 > 50MB

### 稳定性观察项目（如果焦点=稳定性或全部）

在每个检查点，请注意：
- [ ] 自上次检查点以来未发生崩溃、挂起或冻结
- [ ] 帧速率仍在目标预算内 ([target FPS] fps)
- [ ] 音频仍然正常播放（没有不同步或静音）
- [ ] 所有 HUD 元素仍然正确渲染
- [ ] 输入响应符合预期（无输入丢失或滞后尖峰）

### 平衡/疲劳观察项目（如果焦点=平衡或全部）

在每个检查点收集主观观察结果：
- [ ] 核心机制仍然感觉很有意义（Y/N）
- [ ] 感知难度级别：[too easy / appropriate / too hard]
- [ ] 自上次检查点以来是否有过“我以前见过这个”的时刻？ （小说内容耗尽）
- [ ] 自上次检查点以来有过沮丧的时刻吗？注意原因。
- [ ] 自上次检查点以来，有没有出现过参与度最高的时刻？注意原因。

## 协作协议

- **此技能生成一个协议 - 人类运行它** - 永远不要尝试
  自动运行浸泡测试。观察需要人类观察者。
- **持续时间应与游戏的会话设计相匹配** — 5 分钟的游戏
  不需要浸泡4小时；城市建设者可能会。运用判断力，如果不清楚就询问。
- **第一次浸泡应该是 `all` 焦点** - 窄焦点（仅限内存）用于
  回归在特定修复后浸泡，而不是第一次通过
- **写入前询问** — 在创建协议文件之前始终确认
