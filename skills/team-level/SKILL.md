**决策点：** 在每个步骤转换时，使用 `AskUserQuestion` 来呈现
用户将子代理的建议作为可选择的选项。 Write 代理的
在对话中进行全面分析，然后用简洁的标签捕捉决策。
用户必须批准才能进行下一步。

1. **Read 目标级别或区域的参数**（例如，`tutorial`，
   `forest dungeon`、`hub town`、`final boss arena`）。

2. **收集上下文**：
   - Read `design/gdd/game-concept.md` 的游戏概念
   - Read 游戏支柱 `design/gdd/game-pillars.md`
   - Read `design/levels/` 中的现有级别文档
   - Read `design/narrative/` 中的相关叙述文档
   - Read 该地区的世界构建文档 region/faction

## 如何委派

使用 Task 工具将每个团队成员生成为子代理：
- `subagent_type: narrative-director` — 叙事目的、人物、情感弧线
- `subagent_type: world-builder` — 传说背景、环境故事讲述、世界规则
- `subagent_type: level-designer` — 空间布局、节奏、遭遇、导航
- `subagent_type: systems-designer` — 敌人构成、战利品表、难度平衡
- `subagent_type: art-director` — 视觉主题、调色板、照明、资产要求
- `subagent_type: accessibility-specialist` — 导航清晰度、色盲安全、认知负荷
- `subagent_type: qa-tester` — 测试用例、边界测试、游戏测试清单

始终在每个代理的提示中提供完整的上下文（游戏概念、支柱、现有关卡文档、叙述文档）。

3. **按顺序安排关卡设计团队**：

### 第1步：叙事+视觉指导（叙事导演+世界构建者+艺术总监，并行）

同时生成所有三个代理 - 在等待任何结果之前发出所有三个 Task 调用。

生成 `narrative-director` 代理以：
- 定义该区域的叙事目的（这里发生什么故事节拍？）
- 识别关键人物、对话触发点和故事元素
- 指定情感弧线（玩家在进入、离开期间应该有什么感觉？）

生成 `world-builder` 代理以：
- 提供该地区的背景知识（历史、派系存在、生态）
- 定义环境讲故事的机会
- 指定影响该区域游戏玩法的任何世界规则

生成 `art-director` 代理以：
- 为该区域建立视觉主题目标——这些是布局的输入，而不是布局的输出
- 定义该区域的色温和照明氛围（它与相邻区域有何不同？）
- 指定形状语言方向（有棱角的堡垒？有机洞穴？腐烂的宏伟？）
- 命名将引导玩家的主要视觉地标
- Read `design/art/art-bible.md`（如果存在）——将所有方向锚定在既定的艺术圣经中

**第 1 步中艺术总监的视觉目标必须作为显式约束传递给第 2 步中的关卡设计师**。布局决策发生在视觉方向之内，而不是之前。

**门**：使用 `AskUserQuestion` 呈现所有三个步骤 1 输出（叙述简介、背景知识、视觉方向目标）并在继续步骤 2 之前进行确认。

### 第2步：布局和遭遇设计（关卡设计师）
生成 `level-designer` 代理，并将完整的步骤 1 输出作为上下文：
- 叙事简介（来自叙事导演）
- 传说基础（来自世界建造者）
- **视觉方向目标（来自艺术总监）** - 布局必须在这些目标范围内工作，而不是与它们相矛盾

关卡设计师应该：
- 设计空间布局（关键路径、可选路径、秘密）——确保主要路线与步骤 1 中的视觉地标目标保持一致
- 定义节奏曲线（紧张峰值、休息区、探索区）——与叙事导演的情绪弧相协调
- 遇到困难的地方
- 设计环境谜题或导航挑战
- 定义用于寻路的兴趣点和地标——这些必须与艺术总监指定的视觉地标相匹配
- 指定 entry/exit 点以及与相邻区域的连接

**相邻区域依赖性检查**：生成布局后，检查关卡设计者引用的每个相邻区域的 `design/levels/`。如果任何引用区域的 `.md` 文件不存在，则显示间隙：
> “级别引用 [area-name] 作为相邻区域，但 `design/levels/[area-name].md` 不存在。”

使用 `AskUserQuestion` 和选项：
- (a) 继续使用占位符引用 - 在级别文档中将连接标记为“未解决”，并将其列在摘要报告的开放跨级别依赖项部分中
- (b) 首先暂停并运行 `/team-level [area-name]` 以建立该区域

不要为缺失的相邻区域发明内容。

**门**：使用 `AskUserQuestion` 呈现步骤 2 布局（包括任何未解决的相邻区域依赖性）并在继续步骤 3 之前进行确认。

### 第 3 步：系统集成（系统设计师）
生成 `systems-designer` 代理以：
- 指定敌人的构成和遭遇公式
- 定义战利品表和奖励放置
- 相对于预期玩家的平衡难度 level/gear
- 设计任何特定区域的机械或环境危害
- 指定资源分配（生命值拾取、保存点、商店）

**门**：使用 `AskUserQuestion` 呈现步骤 3 的输出并在继续步骤 4 之前进行确认。

### 第 4 步：制作概念 + 辅助功能（艺术总监 + 辅助功能专家，并行）

**注**：艺术总监的定向通道（视觉主题、色彩目标、情绪）发生在步骤 1 中。此通道是特定于位置的制作概念 - 给定最终布局，每个特定空间是什么样子？

使用步骤 2 中的最终布局生成 `art-director` 代理：
- 为关键空间（入口、关键接触区、地标、出口）制定特定位置的概念规范
- 指定哪些艺术资产是该区域独有的，哪些是从全球资源库中共享的
- 定义每个关键空间的视线和照明设置（这些现在是布局通知的，而不是方向性的）
- 指定 VFX 特定于此区域布局的需求（天气量、颗粒物、大气影响）
- 标记布局与步骤 1 目标产生视觉方向冲突的任何位置 - 将这些位置作为生产风险

并行生成 `accessibility-specialist` 代理：
- 检查关卡布局以确保导航清晰度（玩家可以在不单独依赖颜色的情况下确定自己的方向吗？）
- 检查关键路径路标除了颜色之外还使用 shape/icon/sound 提示
- 检查任何谜题机制的认知负荷 — 标记任何需要同时保持 3 个以上状态的内容
- 检查关键游戏区域是否对色盲玩家有足够的对比度
- 输出：具有严重性的可访问性问题列表（阻止/推荐/最好有）

等待两名特工返回后再继续。

**门**：使用 `AskUserQuestion` 呈现第 4 步的结果。如果辅助功能专家返回了任何阻塞问题，请突出显示它们并提供：
- (a) 返回关卡设计师和艺术总监，重新设计步骤 5 之前的标记元素
- (b) 记录为已知的可访问性差距，并继续执行步骤 5，并将问题明确记录在最终报告中

在用户未确认任何阻塞可访问性问题之前，请勿继续执行步骤 5。

### 步骤 5：QA 规划（qa 测试人员）
生成 `qa-tester` 代理以：
- Write 关键路径的测试用例
- 识别边界和边缘情况（序列中断、软锁）
- 为该区域创建游戏测试清单
- 定义级别完成的验收标准

4. **编译关卡设计文档** 将所有团队输出合并到
   关卡设计模板格式。

5. **保存到** `design/levels/[level-name].md`。

6. **输出摘要**：区域概览、遭遇次数、估计资产
   列表、叙述节拍、任何跨团队依赖性或开放性问题，开放
   跨级依赖（引用但尚未设计的相邻区域，每个
   标记为“未解决”），以及对其解决状态的可访问性问题。

## 文件 Write 协议

所有文件写入（关卡设计文档、叙述文档、测试清单）均被委托
通过 Task 生成的子代理。每个子代理强制执行“我可以写入 [path] 吗？”
协议。该编排器不直接写入文件。

结论：**完成**——制作了关卡设计文档并编译了所有团队输出。
结论：**被封锁**——一名或多名特工被封锁；列出未解决项目的部分报告。

## 下一步

- 运行 `/design-review design/levels/[level-name].md` 以验证已完成的关卡设计文档。
- 设计获得批准后，运行 `/dev-story` 以实施关卡内容。
- 运行 `/qa-plan` 以生成此级别的 QA 测试计划。

## 错误恢复协议

如果任何生成的代理（通过 Task）返回 BLOCKED、错误或无法完成：

1. **立即浮出水面**：在继续依赖阶段之前向用户报告“[AgentName]：BLOCKED — [reason]”
2. **评估依赖关系**：检查后续阶段是否需要被阻止的代理的输出。如果是，则在没有用户输入的情况下不要继续越过该依赖点。
3. **通过 AskUserQuestion 提供选项**，可选择：
   - 跳过此代理并注意最终报告中的差距
   - 以更窄的范围重试
   - 停在这里并首先解决阻止程序
4. **始终生成部分报告** — 输出已完成的内容。切勿因为一名代理阻塞而放弃工作。

常见的拦截器：
- 输入文件丢失（未找到故事，GDD 不存在）→ 重定向到创建它的技能
- ADR 状态为建议 → 不实施；首先运行 `/architecture-decision`
- 范围太大 → 通过 `/create-stories` 分成两个故事
- ADR 和故事之间的指令冲突 → 冲突浮出水面，不要猜测
