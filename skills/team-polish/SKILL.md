## 团队构成
- **性能分析师** — 分析、优化、内存分析、帧预算
- **引擎程序员** — 引擎级瓶颈：渲染管道、内存、资源加载（当性能分析师识别出低级根本原因时调用）
- **技术美术师** — VFX 打磨、着色器优化、视觉质量
- **声音设计师** — 音频润色、混合、环境层、反馈声音
- **工具程序员** - 内容管道工具验证、编辑器工具稳定性、自动化修复（打磨区域涉及内容创作工具时调用）
- **qa-tester** — 边缘案例测试、回归测试、浸泡测试

## 如何委派

使用 Task 工具将每个团队成员生成为子代理：
- `subagent_type: performance-analyst` — 分析、优化、内存分析
- `subagent_type: engine-programmer` — 针对渲染、内存、资源加载的引擎级修复
- `subagent_type: technical-artist` — VFX 抛光、着色器优化、视觉质量
- `subagent_type: sound-designer` — 音频抛光、混合、环境层
- `subagent_type: tools-programmer` — 内容管道和编辑器工具验证
- `subagent_type: qa-tester` — 边缘情况测试、回归测试、浸泡测试

始终在每个代理的提示中提供完整的上下文（目标 feature/area、性能预算、已知问题）。在管道允许的情况下并行启动独立代理（例如，第 3 阶段和第 4 阶段可以同时运行）。

## Pipeline

### 第一阶段：评估
委托给**性能分析师**：
- 使用 `/perf-profile` 分析目标 feature/area
- 识别性能瓶颈和框架预算违规情况
- 测量内存使用情况并检查是否存在泄漏
- 针对目标硬件规格进行基准测试
- 输出：带有优先优化列表的性能报告

### 第二阶段：优化
委托给**性能分析师**（根据需要与相关程序员一起）：
- 修复第一阶段确定的性能热点
- 优化绘制调用，减少透支
- 修复内存泄漏并减少分配压力
- 验证优化不会改变游戏行为
- 输出：具有 before/after 指标的优化代码

如果第 1 阶段确定了引擎级根本原因（渲染管道、资源加载、内存分配器），请将这些修复并行委托给 **引擎程序员**：
- 优化发动机系统中的热路径
- 修复核心循环中的分配压力
- 输出：通过分析器验证进行引擎级修复

### 第 3 阶段：视觉打磨（与第 2 阶段并行）
委托给**技术美术人员**：
- 查看 VFX 的质量和与艺术圣经的一致性
- 优化粒子系统和着色器效果
- 在适当的情况下添加屏幕抖动、相机效果和视觉效果
- 确保效果在较低设置下正常降级
- 输出：精美的视觉效果

### 第 4 阶段：音频润色（与第 2 阶段并行）
委托给**声音设计师**：
- 检查音频事件的完整性（是否有任何操作缺少声音反馈？）
- 检查音频混合级别 - 相对于混音，不要太大声或太安静
- 添加环境音频层以营造氛围
- 通过空间定位验证音频是否正确播放
- 输出：音频润色列表和混音笔记

### 第五阶段：强化
委托给 **qa-tester**：
- 测试所有边缘情况：边界条件、快速输入、异常序列
- 浸泡测试：长时间运行该功能以检查性能退化情况
- 压力测试：最大实体数、最坏情况
- 回归测试：验证改进更改没有破坏现有功能
- 在最低规格硬件上进行测试（如果可用）
- 输出：测试结果以及任何剩余问题

### 第 6 阶段：签核
- 收集所有团队成员的结果
- 将绩效指标与预算进行比较
- 报告：已准备好发布/需要更多工作
- 列出所有剩余问题的严重性和建议

## 错误恢复协议

如果任何生成的代理（通过 Task）返回 BLOCKED、错误或无法完成：

1. **立即浮出水面**：在继续依赖阶段之前向用户报告“[AgentName]：BLOCKED — [reason]”
2. **评估依赖关系**：检查后续阶段是否需要被阻止的代理的输出。如果是，则在没有用户输入的情况下不要继续越过该依赖点。
3. **通过 AskUserQuestion 提供选项**，可选择：
   - 跳过此代理并注意最终报告中的差距
   - 以更窄的范围重试
   - 停在这里并首先解决阻止程序
4. **始终生成部分报告** — 输出已完成的内容。切勿因为一名代理阻塞而放弃工作。

常见的拦截器：
- 输入文件丢失（未找到故事，GDD 不存在）→ 重定向到创建它的技能
- ADR 状态为建议 → 不实施；首先运行 `/architecture-decision`
- 范围太大 → 通过 `/create-stories` 分成两个故事
- ADR 和故事之间的指令冲突 → 冲突浮出水面，不要猜测

## 文件 Write 协议

所有文件写入（性能报告、测试结果、证据文档）均委托给
通过 Task 生成的子代理。每个子代理强制执行“我可以写信给 [path] 吗？”
协议。该编排器不直接写入文件。

## Output

摘要报告涵盖：性能 before/after 指标、视觉修饰变化、音频修饰变化、测试结果和发布准备评估。
