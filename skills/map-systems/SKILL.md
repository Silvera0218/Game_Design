## 解析参数

两种模式：

- **无参数**：`/map-systems` — 运行完整的分解工作流程（阶段 1-5）
  创建或更新系统索引。
- **`next`**：`/map-systems next` — 选择最高优先级的未设计系统
  从索引中移出并移交给 `/design-system`（第 6 阶段）。

还解决审查模式（一次，存储本次运行的所有门生成）：
1. 如果 `--review [完整|lean|独奏]`已通过→使用
2. 否则读取 `production/review-mode.txt` → 使用该值
3. 否则 → 默认为 `lean`

有关完整检查模式，请参阅 `.codex/docs/director-gates.md`。

## 第 1 阶段：Read 概念（必需的上下文）

Read 游戏概念和任何现有的设计工作。这就提供了原材料
用于系统分解。

**Required:**
- Read `design/gdd/game-concept.md` — **如果丢失，则会失败并显示明确的消息**：
  > “在 `design/gdd/game-concept.md` 中找不到游戏概念。首先运行 `/brainstorm`
  > 创建一个，然后回来将其分解为系统。”

**可选（如果存在则阅读）：**
- Read `design/gdd/game-pillars.md` — 支柱限制优先级和范围
- Read `design/gdd/systems-index.md` — 如果存在，**从中断处继续**
  （更新，不要从头开始重新创建）
- Glob `design/gdd/*.md` — 检查哪个系统 GDD 已存在

**如果系统索引已经存在：**
- Read 它并向用户呈现当前状态
- 使用`AskUserQuestion`询问：
  “系统索引已存在于 [N] 系统中（[M] 已设计，[K] 未启动）。
  你想做什么？”
  - 选项：“用新系统更新索引”、“设计下一个未设计的系统”、
    “审查并修改优先事项”

## 第 2 阶段：系统枚举（协作）

提取并识别游戏所需的所有系统。这就是该作品的创作核心
技能——它需要人类的判断，因为概念文档很少列举每一个
系统明确。

### 步骤 2a：提取显式系统

扫描游戏概念以获取直接提到的系统和机制：
- 核心机制部分（最明确）
- 核心循环部分（暗示哪些系统驱动每个循环层）
- 技术考虑部分（网络、程序生成等）
- MVP 定义部分（所需功能 = 所需系统）

### 步骤 2b：识别隐式系统

对于每个显式系统，识别它所暗示的**隐藏系统**。游戏总是
需要比概念文档提到的更多的系统。使用这个推理模式：

- “库存”意味着：物品数据库、装备槽位、weight/capacity规则、
  库存 UI，save/load 的物料序列化
- “战斗”意味着：伤害计算、生命系统、命中检测、状态效果、
  敌人AI，战斗UI（生命条，伤害数字），death/respawn
- “开放世界”暗示：streaming/chunking、LOD系统、快速旅行、map/minimap、
  兴趣点跟踪、世界状态持久性
- “多人游戏”意味着：网络层、lobby/matchmaking、状态同步、
  反作弊，网络 UI（ping，玩家列表）
- “制作”意味着：配方数据库、原料收集、制作UI，
  success/failure 力学，配方 discovery/learning
- “对话”意味着：对话树系统、对话UI、选择跟踪、NPC
  状态管理、本地化挂钩
- “进步”意味着：XP系统、升级机制、技能树、解锁
  跟踪、进度 UI、进度保存数据

在对话文本中解释为什么需要每个隐式系统（举例）。

### 步骤 2c：用户审核

呈现按类别组织的枚举。对于每个系统，显示：
- Name
- Category
- 简要说明（1句话）
- 无论是明确的（从概念）还是隐含的（推断）

然后使用 `AskUserQuestion` 捕获反馈：
- “这个列表中是否缺少系统？”
- “这些中的任何一个应该合并还是分开？”
- “是否列出了该游戏不需要的系统？”

迭代直到用户批准枚举。

## 第 3 阶段：依赖关系映射（协作）

对于每个系统，确定它所依赖的内容。一个系统“依赖于”另一个系统
如果没有其他系统首先存在，它就无法运行。

### 步骤 3a：映射依赖关系

对于每个系统，列出其依赖项。使用这些依赖启发式：
- **Input/output 依赖关系**：系统 A 生成系统 B 需要的数据
- **结构依赖**：系统 A 提供系统 B 插入的框架
- **UI 依赖项**：每个游戏系统都有一个相应的 UI 系统
  取决于它（但UI是根据游戏系统设计的）

### 步骤 3b：按依赖顺序排序

将系统分层：
1. **基础**：零依赖的系统（首先设计和构建）
2. **核心**：仅依赖于基金会系统的系统
3. **功能**：系统取决于核心系统
4. **演示**：UI 和包装游戏系统的反馈系统
5. **波兰**：元系统、教程、分析、可访问性

### 步骤 3c：检测循环依赖关系

检查依赖图中的循环。如果发现：
- 向用户突出显示它们
- 提出解决方案（接口抽象、同步设计、打破
  通过定义两个系统之间的契约来循环）

### 步骤 3d：呈现给用户

将依赖关系图显示为分层列表。亮点：
- 任何循环依赖
- 任何“瓶颈”系统（许多其他系统依赖它们 - 这些都是高风险的）
- 任何没有依赖项的系统（叶节点——风险较低，可以后期设计）

使用 `AskUserQuestion` 询问：“这个依赖顺序看起来正确吗？有吗？
我缺少或应该删除的依赖项？”

**审查模式检查** — 在生成 TD-SYSTEM-BOUNDARY 之前应用：
- `solo` → 跳过。注意：“TD-SYSTEM-BOUNDARY 已跳过 — 单人模式。”继续进行优先级分配。
- `lean` → 跳过（不是相位门）。注意：“TD-SYSTEM-BOUNDARY 已跳过 — 精益模式。”继续进行优先级分配。
- `full` → 正常生成。

**依赖关系映射获得批准后，在继续进行优先级分配之前，使用门 TD-SYSTEM-BOUNDARY (`.codex/docs/director-gates.md`) 通过 Task 生成 `technical-director`。**

通过：依赖关系图摘要、层分配、瓶颈系统列表、任何循环依赖关系解析。

提出评估。如果拒绝，请在进行优先级分配之前与用户一起修改系统边界。如果有疑问，请在系统索引中记下它们并继续。

## 第 4 阶段：优先级分配（协作）

根据每个系统所需的里程碑，将其分配到优先级。

### 步骤 4a：根据概念自动分配

使用这些启发式进行初始分配：
- **MVP**：概念的“MVP 所需”部分中提到的系统，以及它们的
  基础层依赖
- **垂直切片**：在一个领域获得完整体验所需的系统
- **Alpha**：所有剩余的游戏系统
- **Full Vision**：完善、元和锦上添花的系统

### 步骤 4b：用户审核

在表格中呈现优先级分配。对于每一层，解释为什么系统
被放置在那里。

使用 `AskUserQuestion` 询问：“这些优先级分配符合您的愿景吗？
哪些系统应该具有更高或更低的优先级？”

在对话中解释推理：“我将 [system] 放在 MVP 中，因为核心循环
需要它——没有 [system]，30 秒循环就无法运行。”

**“原因”专栏指南**：在解释为什么每个系统被置于优先级时，将技术必要性与玩家体验推理相结合。不要使用纯粹的技术理由，例如“战斗需要伤害数学”——在相关的情况下与玩家体验相关。好的“为什么”条目示例：
- “核心循环所必需的 - 没有它，布局决策就没有任何后果（支柱 2：布局是难题）”
- “弩车的穿透特性在这里建立 - 这个统计定义使它感觉与弓箭手不同”
- “所有经济决策的基础——玩家必须了解升级成本才能做出有意义的放置选择”

当系统直接塑造玩家体验时，单纯的技术必要性（“X 取决于 Y”）是不够的。

**审查模式检查** — 在生成 PR-SCOPE 之前应用：
- `solo` → 跳过。注意：“PR-SCOPE 已跳过 — 独奏模式。”继续编写系统索引。
- `lean` → 跳过（不是相位门）。注意：“PR-SCOPE 已跳过 — 精益模式。”继续编写系统索引。
- `full` → 正常生成。

**优先级获得批准后，在写入索引之前，使用门 PR-SCOPE (`.codex/docs/director-gates.md`) 通过 Task 生成 `producer`。**

通过：每个里程碑层的系统总数、每个层的预计实施量（系统数量 × 平均复杂度）、团队规模、规定的项目时间表。

提出评估。如果不现实，请在写入索引之前提出修改优先级分配。如果有疑虑，请记下并继续。

### 步骤 4c：确定设计顺序

结合依赖排序+优先级来产生最终的设计顺序：
1. MVP 基础系统第一
2. MVP 核心系统第二
3. MVP 特征系统第三
4. 垂直切片 Foundation/Core 系统
5. ...and so on

这是团队编写 GDD 的顺序。

## 第 5 阶段：创建系统索引 (Write)

### 步骤 5a：起草文档

使用 `.codex/docs/templates/systems-index.md` 中的模板，填充
包含第 2-4 阶段所有数据的系统索引：
- 填写枚举表
- 填写依赖关系图
- 填写推荐的设计订单
- 填充高风险系统
- 填充进度跟踪器（所有系统最初“未启动”，除非 GDD 已存在）

### 步骤 5b：批准

呈现该文件的摘要：
- 按类别划分的系统总数
- MVP 系统计数
- 设计顺序中的前 3 个系统
- 任何高风险物品

问：“我可以将系统索引写入`design/gdd/systems-index.md`吗？”

等待批准。 Write 仅在“是”之后才显示该文件。

**查看模式检查** — 在生成 CD-SYSTEMS 之前应用：
- `solo` → 跳过。注意：“已跳过 CD-SYSTEMS — 单人模式。”继续进行第 7 阶段的后续步骤。
- `lean` → 跳过（不是相位门）。注意：“已跳过 CD-SYSTEMS — 精益模式。”继续进行第 7 阶段的后续步骤。
- `full` → 正常生成。

**写入系统索引后，使用门 CD-SYSTEMS (`.codex/docs/director-gates.md`) 通过 Task 生成 `creative-director`。**

通行证：系统索引路径、游戏支柱和核心幻想（来自`design/gdd/game-concept.md`）、MVP优先级系统列表。

提出评估。如果拒绝，请在 GDD 创作开始之前与用户一起修改系统设置。如果有疑虑，请将其记录在系统索引中相关层部分顶部的 `> **Creative Director Note**` 中。

### 步骤 5c：更新会话状态

写入后，如果 `production/session-state/active.md` 不存在，则创建它，然后更新为：
- Task：系统分解
- 状态：系统索引已创建
- 文件：design/gdd/systems-index.md
- 下一步：设计单独的系统 GDD

**结论：完整** — 系统索引写入 `design/gdd/systems-index.md`。
如果用户拒绝： **结论：被阻止** — 用户未批准写入。

## 第 6 阶段：设计单独的系统（移交给 /design-system）

当出现以下情况时进入该阶段：
- 用户在创建索引后对系统设计说“是”
- 用户调用`/map-systems [system-name]`
- 用户调用`/map-systems next`

### 步骤 6a：选择系统

- 如果提供了系统名称，请在系统索引中查找它
- 如果使用 `next`，则选择最高优先级的未设计系统（按设计顺序）
- 如果用户刚刚完成索引，则询问：
  “您现在想开始设计单独的系统吗？第一个系统
  设计订单为[name]。还是你想在这里停下来，稍后再回来？”

使用 `AskUserQuestion`：“现在开始设计 [system-name]，选择一个不同的
系统，还是停在这里？”

### 步骤 6b：移交给 /design-system

选择系统后，调用 `/design-system [system-name]` 技能。

`/design-system` 技能处理完整的 GDD 创作过程：
- 从游戏概念、系统索引和依赖 GDD 中收集上下文
- 立即创建文件骨架
- 一次浏览所有 8 个必需部分（协作、增量）
- 交叉引用现有文档以防止矛盾
- 获取领域专业知识的专业代理的路线
- 一旦获得批准，将每个部分写入文件
- 完成后运行 `/design-review`
- 更新系统索引

**请勿在此处重复 /design-system 工作流程。** 此技能拥有系统
*索引*； `/design-system` 拥有单独的系统*GDD*。

### 步骤 6c：循环或停止

`/design-system`完成后，使用`AskUserQuestion`：
- “继续下一个系统（[next system name]）？”
- “选择不同的系统？”
- “这次会议就停在这里吗？”

如果继续，请返回步骤 6a。

## 第 7 阶段：建议后续步骤

创建系统索引后（或设计某些系统后），使用 `AskUserQuestion` 呈现下一步操作：

- “系统索引已写入。接下来您想做什么？”
  - [A] 开始设计 GDD — 运行 `/design-system [first-system-in-order]`
  - [B] 要求主管首先检查索引 — 要求 `creative-director` 或 `technical-director` 在提交 10 多个 GDD 会话之前验证系统集
  - [C] 本次会议在此停止

**导演审查选项 ([B]) 值得强调**：在开始 GDD 创作之前让创意总监或技术总监审查已完成的系统索引，可以在范围问题、缺失系统和边界问题被锁定在许多文档中之前捕获它们。它是可选的，但建议用于新项目。

完成任何单个 GDD 后：
- “在新会话中运行 `/design-review design/gdd/[system].md` 以验证质量”
- “当所有 MVP GDD 完成时运行 `/gate-check systems-design`”

## 协作协议

该技能在每个阶段都遵循协作设计原则：

1. **问题 -> 选项 -> 决定 -> 草案 -> 批准** 每一步
2. **AskUserQuestion** 在每个决策点（解释 -> 捕获模式）：
   - 第二阶段：“缺失的系统？合并还是分裂？”
   - 第 3 阶段：“依赖顺序正确吗？”
   - 第四阶段：“优先任务符合你的愿景吗？”
   - 阶段 5：“我可以写系统索引吗？”
   - 第六阶段：“开始设计，选择不同的，还是停止？”然后交给 `/design-system`
3. **“我可以写入 [filepath] 吗？”** 在每次写入文件之前
4. **增量写入**：每个系统设计完成后更新系统索引
5. **交接**：个人 GDD 创作归 `/design-system` 所有，负责处理
   增量章节编写、交叉引用、设计审查和索引更新
6. **会话状态更新**：Write 到 `production/session-state/active.md` 之后
   每个里程碑（创建索引、设计系统、更改优先级）

**永远不要**自动生成完整的系统列表并在未经审查的情况下编写它。
**永远不要**在未经用户确认的情况下开始设计系统。
**始终** 显示用户验证的枚举、依赖关系和优先级。
