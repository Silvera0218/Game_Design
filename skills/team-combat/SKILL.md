## 团队构成
- **游戏设计师** — 设计机制、定义公式和边缘情况
- **游戏程序员** — 实现核心游戏代码
- **ai-programmer** — 为该功能实现 NPC/enemy AI 行为
- **技术美术师** — 创建 VFX、着色器效果和视觉反馈
- **声音设计器** — 定义音频事件、冲击声音和环境战斗音频
- **引擎专家**（主要）- 验证架构和实现模式对于引擎来说是否惯用（阅读 `.codex/docs/technical-preferences.md` 引擎专家部分）
- **qa-tester** — Write 测试用例并验证实施

## 如何委派

使用 Task 工具将每个团队成员生成为子代理：
- `subagent_type: game-designer` — 设计机制、定义公式和边缘情况
- `subagent_type: gameplay-programmer` — 实现核心游戏代码
- `subagent_type: ai-programmer` — 实现 NPC/enemy AI 行为
- `subagent_type: technical-artist` — 创建 VFX、着色器效果、视觉反馈
- `subagent_type: sound-designer` — 定义音频事件、冲击声、环境音频
- `subagent_type: [primary engine specialist]` — 架构和实现的引擎习语验证
- `subagent_type: qa-tester` — Write 测试用例并验证实施

始终在每个代理的提示中提供完整的上下文（设计文档路径、相关代码文件、约束）。在管道允许的情况下并行启动独立代理（例如，第 3 阶段代理可以同时运行）。

## Pipeline

### 第一阶段：设计
委托给**游戏设计师**：
- 创建或更新 `design/gdd/` 中的设计文档，涵盖：机制概述、玩家幻想、详细规则、具有变量定义的公式、边缘情况、依赖性、具有安全范围的调节旋钮以及验收标准
- 输出：完成的设计文件

### 第二阶段：架构
委托给 **gameplay-programmer**（如果涉及 AI，则使用 **ai-programmer**）：
- 审查设计文件
- 设计代码架构：类结构、接口、数据流
- 确定与现有系统的集成点
- 输出：带有文件列表和接口定义的架构草图

然后派生**主要引擎专家**来验证建议的架构：
- class/node/component 结构对于固定发动机来说是惯用的吗？ （例如，Godot 节点层次结构、Unity MonoBehaviour 与 DOTS、Unreal Actor/Component 设计）
- 是否应该使用引擎本机系统来代替自定义实现？
- 在固定引擎版本中是否有任何建议的 API 已弃用或更改？
- 输出：引擎架构注释 - 在第 3 阶段开始之前合并到架构中

### 第 3 阶段：实施（尽可能并行）
并行委托：
- **游戏程序员**：实现核心战斗机制代码
- **ai-programmer**：实现 AI 行为（如果该功能涉及 NPC 反应）
- **技术艺术家**：创建 VFX 和着色器效果
- **声音设计器**：定义音频事件列表和混合音符

### 第四阶段：整合
- 将游戏代码、AI、VFX 和音频连接在一起
- 确保所有调谐旋钮均公开且由数据驱动
- 验证该功能是否适用于现有的战斗系统

### 第五阶段：验证
委托给 **qa-tester**：
- Write 验收标准中的测试用例
- 测试设计中记录的所有边缘情况
- 验证性能影响是否在预算范围内
- 针对发现的任何问题提交错误报告

### 第 6 阶段：签核
- 收集所有团队成员的结果
- 报告功能状态：已完成/需要工作/已阻止
- 列出所有未决问题及其指定的所有者

## 错误恢复协议

如果任何生成的代理（通过 Task）返回 BLOCKED、错误或无法完成：

1. **立即浮出水面**：在继续依赖阶段之前向用户报告“[AgentName]：BLOCKED — [reason]”
2. **评估依赖关系**：检查后续阶段是否需要被阻止的代理的输出。如果是，则在没有用户输入的情况下不要继续越过该依赖点。
3. **通过 AskUserQuestion 提供选项**，可选择：
   - 跳过此代理并注意最终报告中的差距
   - 以更窄的范围重试
   - 停在这里并首先解决阻止程序
4. **始终生成部分报告** — 输出已完成的内容。切勿因为一名代理阻塞而放弃工作。

常见的拦截器：
- 输入文件丢失（未找到故事，GDD 不存在）→ 重定向到创建它的技能
- ADR 状态为建议 → 不实施；首先运行 `/architecture-decision`
- 范围太大 → 通过 `/create-stories` 分成两个故事
- ADR 和故事之间的指令冲突 → 冲突浮出水面，不要猜测

## 文件 Write 协议

所有文件写入（设计文档、实现文件、测试用例）
委托给通过 Task 生成的子代理。每个子代理强制执行
“我可以写信给[path]吗？”协议。该编排器不直接写入文件。

## Output

摘要报告涵盖：设计完成状态、每个团队成员的实施状态、测试结果以及任何未解决的问题。

结论：**完整**——战斗功能的设计、实施和验证。
结论： **受阻** — 一个或多个阶段无法完成；列出未解决项目的部分报告。
