当该技能被调用时：

1. **解析参数**以获得可选的 genre/theme 提示（例如，`roguelike`，
   `space survival`、`cozy farming`）。如果 `open` 或无参数，则从
   刮擦。还解决审查模式（一次，存储本次运行的所有门生成）：
   1. 如果`--review [完整|lean|独奏]`已通过→使用
   2. 否则读取 `production/review-mode.txt` → 使用该值
   3. 否则 → 默认为 `lean`

   有关完整检查模式，请参阅 `.codex/docs/director-gates.md`。

2. **检查现有的概念工作**：
   - Read `design/gdd/game-concept.md` 如果存在（恢复，不重新启动）
   - Read `design/gdd/game-pillars.md` 如果存在（建立在已建立的支柱上）

3. **以交互方式运行构思阶段**，在以下位置询问用户问题
每个阶段。不要默默地生成所有内容 - 目标是**协作
   探索**，人工智能充当创意促进者，而不是替代者
   为了人类的视觉。

   **在整个头脑风暴过程中的关键决策点使用 `AskUserQuestion`**：
   - 受限品味问题（类型偏好、范围、团队规模）
   - 提出选项后概念选择（“哪 2-3 个概念引起共鸣？”）
   - 方向选择（“进一步开发、探索更多，还是原型？”）
   - 概念细化后的支柱排名
   Write 先对对话文本进行全面创意分析，然后使用
   `AskUserQuestion` 使用简洁的标签捕获决策。

   专业工作室头脑风暴应遵循的原则：
   - 保留判断——在探索过程中没有什么想法是坏的
   - 鼓励不寻常的想法——打破常规的思维激发更好的概念
   - 相互借鉴——回答“是的，并且……”，而不是“但是……”
   - 使用限制作为创意燃料——限制往往会产生最好的想法
   - 每个阶段都有时间限制——保持动力，不要过早过度考虑

### 第一阶段：创意发现

首先要了解人，而不是游戏。问这些问题
对话方式（不是清单）：

**情感锚**：
- 游戏中的哪个时刻真正让你感动、激动，或者让你
  你忘记了时间吗？具体是什么造成了这种感觉？
- 游戏中有没有你一直想要但从未实现的幻想或权力之旅
  完全找到了？

**口味简介**：
- 您花最多时间玩哪 3 款游戏？是什么让你回来？
  *（以纯文本形式询问 - 用户必须能够自由输入特定的游戏名称。
  请勿将其放入带有预设选项的 AskUserQuestion 中。）*
- 有你喜欢的类型吗？您回避的流派？为什么？
- 你喜欢那些有挑战性、让你放松、给你讲故事的游戏吗？
  或者让你表达自己？ *（为此使用 `AskUserQuestion` — 受限选择。）*

**实际限制**（在集思广益之前塑造沙箱）。
使用这些确切的选项卡标签将它们捆绑到单个多选项卡 `AskUserQuestion` 中：
- “体验”选项卡——“你最希望玩家拥有什么样的体验？” （挑战与掌握/故事与发现/表达与创造力/放松与流动）
- “时间表”选项卡 — “您的实际开发时间表是多少？” （周/月/1-2 年/多年）
- 选项卡“开发级别”——“您的开发之旅处于哪个阶段？” （第一款游戏/之前发货/专业背景）

准确使用这些选项卡名称 - 不要重命名或重复它们。

**将答案综合**成**创意简报** — 3-5 句话
对人的情感目标、品味概况和约束的总结。
Read 简短返回并确认它捕获了他们的意图。

### 第二阶段：概念生成

以创意简报为基础，生成 **3 个不同的概念**
每个人都采取不同的创作方向。使用这些构思技巧：

**技术 1：动词优先设计**
从核心玩家动词开始（构建、战斗、探索、解决、生存、
创建、管理、发现）并从那里向外构建。动词就是游戏。

**技术2：混搭方法**
结合两个意想不到的元素：[Genre A] + [Theme B]。之间的紧张关系
两人创造了独特的钩子。 （例如，“农业模拟 + 宇宙恐怖”，
“roguelike + 约会模拟”、“城市建设 + 实时战斗”）

**技术 3：体验优先设计（MDA 向后）**
从期望的玩家情感出发（来自MDA框架的审美目标：
感觉、幻想、叙事、挑战、友谊、发现、表达、
提交）并回溯到产生它的动力学和机制。

对于每个概念，呈现：
- **工作名称**
- **电梯推介**（1-2 句话 — 必须通过“10 秒测试”）
- **核心动词**（最常见的玩家动作）
- **核心幻想**（情感承诺）
- **独特的 Hook**（通过“并且还”测试：“Like X, AND ALSO Y”）
- **主要 MDA 审美**（哪种情感占主导地位？）
- **估计范围**（小/中/大）
- **为什么它可以工作**（market/audience 适合 1 句话）
- **最大的风险**（关于最难回答的问题的 1 句话）

呈现所有三个。然后使用 `AskUserQuestion` 捕获选择。

**关键**：这必须是一个简单的列表调用 - 没有选项卡，没有表单字段。完全使用这个结构：

```
AskUserQuestion(
  prompt: "Which concept resonates with you? You can pick one, combine elements, or ask for fresh directions.",
  options: [
    "Concept 1 — [Title]",
    "Concept 2 — [Title]",
    "Concept 3 — [Title]",
    "Combine elements across concepts",
    "Generate fresh directions"
  ]
)
```

请勿在此处使用 `tabs` 字段。 `tabs` 表单仅用于多字段输入 - 此处使用它会导致“工具参数无效”错误。这是一个简单的 `prompt` + `options` 调用。

永远不要对选择施加压力——让他们接受它。

### 第三阶段：核心循环设计

对于所选概念，使用结构化提问来构建核心循环。
核心循环是游戏的核心——如果它不好玩的话
孤立，再多的内容或润色也无法拯救游戏。

**30 秒循环**（时刻到时刻）：

以 `AskUserQuestion` 调用的方式询问这些 — 从所选概念中派生选项，不要对它们进行硬编码：

1. **核心动作感觉** — 提示：“核心动作的主要感觉是什么？”生成 3-4 个适合概念流派和基调的选项，以及自由文本转义 (`I'll describe it`)。

2. **关键设计维度** - 确定该特定概念最重要的设计变量（例如，世界反应性、节奏、玩家代理）并询问它。生成与概念相匹配的选项。始终包含自由文本转义。

获得答案后，​​分析：这个行为本质上令人满意吗？是什么让它感觉良好？ （音频反馈、视觉效果、时机满意度、战术深度？）

**5 分钟循环**（短期目标）：
- 是什么将每时每刻的比赛组织成循环？
- “再转一圈”/“再跑一圈”的心理在哪里发挥作用？
- 玩家在这个关卡会做出什么选择？

**会话循环**（30-120 分钟）：
- 完整的会话是什么样的？
- 自然停靠点在哪里？
- 让他们在不玩游戏时思考游戏的“魅力”是什么？

**进展循环** (days/weeks)：
- 玩家如何成长？ （力量？知识？选项？故事？）
- 长期目标是什么？游戏什么时候“完成”？

**玩家动机分析**（基于自我决定理论）：
- **自主性**：玩家有多少有意义的选择？
- **能力**：玩家如何感觉自己的技能在增长？
- **相关性**：玩家如何感觉有联系（与角色、
  其他玩家，还是世界）？

### 第四阶段：支柱和边界

游戏支柱由真正的 AAA 工作室使用（《战神》、《哈迪斯》、《最后的世界》）
我们）让数百名团队成员做出的决策都指向了
相同的方向。即使对于独立开发人员来说，支柱也可以防止范围蔓延和
保持视野敏锐。

协作定义 **3-5 个支柱**：
- 每个支柱都有一个**名称**和**一句话定义**
- 每个支柱都有一个**设计测试**：“如果我们在 X 和 Y 之间进行辩论，
  这个支柱说我们选择__”
- 支柱应该感觉彼此之间产生了紧张感——如果有的话
  支柱指向同一个方向，他们没有做足够的工作

然后定义 **3+ 反支柱**（这个游戏不是）：
- 反支柱可以防止最常见形式的范围蔓延：“不是吗？
  如果……就很酷”的功能不服务于核心愿景
- 框架为：“我们不会做 [thing]，因为它会损害 [pillar]”

**支柱确认**：呈现完整支柱组后，使用 `AskUserQuestion`：
- 提示：“这些支柱适合你的游戏吗？”
- 选项：`[A] Lock these in` / `[B] Rename or reframe one` / `[C] Swap a pillar out` / `[D] Something else`

如果用户选择 B、C 或 D，请进行修改，然后再次使用 `AskUserQuestion`：
- 提示：“支柱已更新。准备好锁定它们了吗？”
- 选项：`[A] Lock these in` / `[B] Revise another pillar` / `[C] Something else`

重复此操作，直到用户选择 [A] 锁定这些内容。

**查看模式检查** — 在生成 CD-PILLARS 和 AD-CONCEPT-VISUAL 之前应用：
- `solo` → 两者都跳过。注意：“CD-PILLARS 已跳过 — 单人模式。AD-CONCEPT-VISUAL 已跳过 — 单人模式。”进入第 5 阶段。
- `lean` → 跳过两者（不是相位门）。注意：“CD-PILLARS 已跳过 — 精益模式。AD-CONCEPT-VISUAL 已跳过 — 精益模式。”进入第 5 阶段。
- `full` → 正常生成。

**在支柱和反支柱达成一致后，在进入第 5 阶段之前，通过 Task 并行生成 `creative-director` 和 `art-director`。同时发出两个 Task 调用 - 不要在启动另一个调用之前等待一个调用。**

- **`creative-director`** — 门 **CD-柱** (`.codex/docs/director-gates.md`)
  通过：完整的柱子集，设计测试，反柱子，核心幻想，独特的钩子。

- **`art-director`** — 门 **AD-CONCEPT-VISUAL** (`.codex/docs/director-gates.md`)
  通过：游戏概念电梯推介、包含设计测试的完整支柱、目标平台（如果已知）、用户提到的任何参考游戏或视觉试金石。

收集两个判决，然后使用双选项卡 `AskUserQuestion` 将它们一起呈现：
- 选项卡**“支柱”**：呈现创意总监的反馈。选项反映了标准 CD-PILLARS 处理 — `Lock in as-is` / `Revise [specific pillar]` / `Discuss further`。
- 选项卡**“视觉锚点”**：呈现艺术总监的 2-3 个指定视觉方向选项。选项：每个命名方向（每个选项一个）+ `Combine elements across directions` + `Describe my own direction`。

用户选择的视觉锚点（命名方向或自定义描述）被存储为**视觉识别锚点** - 它将被写入游戏概念文档并成为艺术圣经的基础。

如果创意总监对支柱返回“担忧”或“拒绝”，请在要求选择视觉锚点之前解决支柱问题——视觉方向应来自已确认的支柱。

### 阶段 5：玩家类型验证

使用 Bartle 分类法和 Quantic Foundry 动机模型，验证
这个游戏实际上是为谁准备的：

- **主要玩家类型**：谁会喜欢这个游戏？ （成就者、探索者、
  社交家、竞争对手、创作者、讲故事的人）
- **二次上诉**：还有谁可能喜欢它？
- **这不适合谁**：明确谁不喜欢这个游戏就像
  重要的是知道谁会
- **市场验证**：是否有提供类似服务的成功游戏
  玩家类型？我们可以从他们的受众规模中了解到什么？

### 第 6 阶段：范围和可行性

将概念落实到现实中：

- **目标平台**：使用 `AskUserQuestion` —“您针对此游戏定位哪些平台？”
  选项：`PC (Steam / Epic)` / `Mobile (iOS / Android)` / `Console` / `Web / Browser` / `Multiple platforms`
  记录答案 - 它直接形成引擎推荐并将传递到 `/setup-engine`。
  如果相关，请注意平台影响（例如，移动设备意味着 Unity 是强烈首选；控制台意味着 Godot 有限制；Web 意味着 Godot 可以干净地导出）。

- **引擎经验**：使用 `AskUserQuestion` —“您已经有一个正在使用的引擎了吗？”
选项：`Godot` / `Unity` / `Unreal Engine 5` / `No preference — help me decide`
  - 如果他们选择了一个引擎→将其记录为他们的偏好并继续前进。不要事后猜测。
  - 如果“无偏好”→ 告诉他们：“在本次会议之后运行 `/setup-engine` — 它将引导您根据您的概念和平台目标做出完整的决策。”不要在这里提出建议。
- **艺术管道**：艺术风格是什么，劳动强度如何？
- **内容范围**：估计 level/area 数量、项目数量、游戏时间
- **MVP 定义**：测试“是
  核心循环有趣吗？”
- **最大风险**：技术风险、设计风险、市场风险
- **范围等级**：完整的愿景与时间耗尽后的船舶是什么？

**审查模式检查** — 在生成之前应用 TD 可行性：
- `solo` → 跳过。注意：“TD-FEASIBILITY 已跳过 — 单人模式。”直接进入范围层定义。
- `lean` → 跳过（不是相位门）。注意：“TD-FEASIBILITY 已跳过 — 精益模式。”直接进入范围层定义。
- `full` → 正常生成。

**确定最大的技术风险后，在定义范围层之前，使用门 TD-FEASIBILITY (`.codex/docs/director-gates.md`) 通过 Task 生成 `technical-director`。**

通过：核心循环描述、平台目标、引擎选择（或“未定”）、已识别的技术风险列表。

向用户呈现评估。如果风险较高，请在最终确定之前重新审视范围。如果有疑虑，请记下并继续。

**审查模式检查** — 在生成 PR-SCOPE 之前应用：
- `solo` → 跳过。注意：“PR-SCOPE 已跳过 — 独奏模式。”继续生成文档。
- `lean` → 跳过（不是相位门）。注意：“PR-SCOPE 已跳过 — 精益模式。”继续生成文档。
- `full` → 正常生成。

**定义范围层后，使用门 PR-SCOPE (`.codex/docs/director-gates.md`) 通过 Task 生成 `producer`。**

通过：完整愿景范围、MVP 定义、时间线估算、团队规模。

向用户呈现评估。如果不现实，请在编写文档之前调整 MVP 定义或范围层。
