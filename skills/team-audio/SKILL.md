## 如何委派

使用 Task 工具将每个团队成员生成为子代理：
- `subagent_type: audio-director` — 声音特性、情感基调、音频调色板
- `subagent_type: sound-designer` — SFX 规范、音频事件、混合组
- `subagent_type: technical-artist` — 音频中间件、总线结构、内存预算
- `subagent_type: [primary engine specialist]` — 验证引擎的音频集成模式
- `subagent_type: gameplay-programmer` — 音频管理器、游戏触发器、自适应音乐

始终在每个代理的提示中提供完整的上下文（功能描述、现有音频资产、设计文档参考）。

3. **按顺序安排音频团队**：

### 第 1 步：音频方向（音频导演）
生成 `audio-director` 代理以：
- 定义此 feature/area 的声音标识
- 指定情绪基调和音频调色板
- 设置音乐方向（自适应层、主干、转场）
- 定义音频优先级和混合目标
- 建立任何自适应音频规则（战斗强度、探索、紧张）

### 第 2 步：声音设计和音频辅助功能（并行）
生成 `sound-designer` 代理以：
- 为每个音频事件创建详细的 SFX 规范
- 定义声音类别（环境、UI、游戏玩法、音乐、对话）
- 指定每个声音的参数（音量范围、音调变化、衰减）
- 规划具有触发条件的音频事件列表
- 定义混合组和闪避规则

并行生成 `accessibility-specialist` 代理：
- 确定哪些音频事件携带关键的游戏信息（受到的伤害、附近的敌人、目标完成），并需要为听力受损的玩家提供视觉替代方案
- 指定字幕要求：哪些音频事件需要字幕、什么文本格式、屏幕持续时间
- 检查游戏状态是否仅通过音频传达（所有状态都必须有视觉后备）
- 查看音频事件列表，看看是否有任何可能给听觉敏感的玩家带来问题的事件（高频警报、突然的响亮事件）
- 输出：集成到音频事件规范中的音频可访问性要求列表

### 步骤 3：技术实施（并行）
生成 `technical-artist` 代理以：
- 设计音频中间件集成 (Wwise/FMOD/native)
- 定义音频总线结构和路由
- 指定每个平台音频资产的内存预算
- 计划流媒体与预加载资产策略
- 设计任何音频反应视觉效果

并行生成**主要引擎专家**（来自 `.codex/docs/technical-preferences.md` 引擎专家）以验证集成方法：
- 提议的音频中间件集成对于引擎来说是否惯用？ （例如，Godot 的内置 AudioStreamPlayer 与 FMOD、Unity 的 Audio Mixer 与 Wwise、Unreal 的 MetaSounds 与 FMOD）
- 应该使用任何特定于引擎的音频 node/component 模式吗？
- 固定引擎版本中已知的音频系统变化会影响集成计划吗？
- 输出：引擎音频集成说明与技术艺术家的计划合并

如果没有配置引擎，则跳过专家生成。

### 第 4 步：代码集成（游戏程序员）
生成 `gameplay-programmer` 代理以：
- 实施音频管理系统或审查现有的
- 将音频事件连接到游戏触发器
- 实施自适应音乐系统（如果指定）
- 设置音频 occlusion/reverb 区域
- Write 音频事件触发器的单元测试

4. **结合所有团队输出来编译音频设计文档**。

5. **保存到** `design/gdd/audio-[feature].md`。

6. **输出摘要**：音频事件计数、估计资产计数、
   实施任务以及团队成员之间的任何悬而未决的问题。

结论：**完成**——音频设计文档已制作完毕，团队流程已完成。

如果管道因依赖项未解决而停止（例如，关键的可访问性差距或用户未解决缺少 GDD）：

结论：**被阻止** — [reason]

## 文件 Write 协议

所有文件写入（音频设计文档、SFX 规范、实现文件）均被委托
通过 Task 生成的子代理。每个子代理都强制执行“我可以写入 [path] 吗？”
协议。该编排器不直接写入文件。

## 下一步

- 在实施开始之前，与音频总监一起审查音频设计文档。
- 设计获得批准后，使用 `/dev-story` 实施音频管理器和事件系统。
- 创建音频资产后运行 `/asset-audit` 以验证命名和格式合规性。

## 错误恢复协议

如果任何生成的代理（通过 Task）返回 BLOCKED、错误或无法完成：

1. **立即浮出水面**：在继续依赖阶段之前向用户报告“[AgentName]：BLOCKED — [reason]”
2. **评估依赖关系**：检查后续阶段是否需要被阻止的代理的输出。如果是，则在没有用户输入的情况下不要继续越过该依赖点。
3. **通过 AskUserQuestion 提供选项**，可选择：
   - 跳过此代理并注意最终报告中的差距
   - 以更窄的范围重试
   - 停在这里并首先解决阻止程序
4. **始终生成部分报告** — 输出已完成的内容。切勿因为一名代理阻塞而放弃工作。

常见的拦截器：
- 输入文件丢失（未找到故事，GDD 不存在）→ 重定向到创建它的技能
- ADR 状态为建议 → 不实施；首先运行 `/architecture-decision`
- 范围太大 → 通过 `/create-stories` 分成两个故事
- ADR 和故事之间的指令冲突 → 冲突浮出水面，不要猜测
