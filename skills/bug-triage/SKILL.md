# 错误分类

该技能将未解决的 bug 积压处理为优先的、冲刺分配的
行动清单。它区分**严重性**（影响有多严重？）和
**优先级**（我们必须多紧急地解决它？），检测系统趋势，以及
确保冲刺之间不会丢失任何关键错误。

**输出：** `production/qa/bug-triage-[date].md`

**何时运行：**
- Sprint 开始 — 将未解决的 bug 分配给新的 sprint 或待办事项列表
- `/team-qa` 完成并提交新错误后
- 当错误数量超过 10 个以上未清项目时

---

## 1. 解析参数

**Modes:**
- `/bug-triage sprint` — 针对当前冲刺进行分类；分配可修复的错误
  冲刺待办事项列表；推迟其余的
- `/bug-triage full` — 无论冲刺范围如何，都会对所有错误进行全面分类
- `/bug-triage trend` — 仅趋势分析（无分配）；只读报告
- 无参数 - 如果当前 sprint 存在，则运行 sprint 模式，否则运行完整模式

---

## 2. 加载 Bug 积压

### 步骤 2a — 发现错误文件

Glob 用于按优先顺序排列的错误报告：
1. `production/qa/bugs/*.md` — 单独的错误报告文件（首选格式）
2. `production/qa/bugs.md` — 单个综合错误日志（后备）
3. 任何 `production/qa/qa-plan-*.md`“发现错误”表（最后的手段）

如果没有发现错误文件：
> “在 `production/qa/bugs/` 中没有发现错误文件。如果在
> 不同的位置，调整 glob 模式。如果还不存在错误，则有
> 没有什么可以分类的。”

停下来报告。如果不存在错误，请不要继续。

### 步骤 2b — 加载 sprint 上下文

Read 最近修改的文件 `production/sprints/` 了解：
- 当前冲刺编号/名称
- 范围内的故事（用于分配目标）
- Sprint 容量限制（如果注明）

如果不存在冲刺文件：请注意“未找到冲刺计划 - 仅分配给待办事项”。

### 步骤 2c — 负载严重性参考

Read `.codex/docs/coding-standards.md` 对于 severity/priority 定义，如果它们
存在。如果它们不存在，请使用步骤 3 中的标准定义。

---

## 3.对每个Bug进行分类

对于每个错误，提取或推断：

### 严重性（错误的影响）

| Severity | Definition |
|----------|-----------|
| **S1 — 关键** | 游戏崩溃、数据丢失或功能完全失败。无法继续越过这一点。 |
| **S2 — 高** | 主要功能已损坏，但游戏仍然可以玩。重大错误行为。 |
| **S3 — 中** | 功能下降，但存在解决方法。轻微的错误行为。 |
| **S4 — 低** | 视觉故障、外观问题、打字错误。没有游戏影响。 |

### 优先级（修复的紧急程度）

| Priority | Definition |
|----------|-----------|
| **P1 — 修复此冲刺** | 阻止 QA，阻止发布，或者是上次冲刺的回归 |
| **P2 — 尽快修复** | 应在下一个重大里程碑之前解决 |
| **P3 — 待办事项** | 很好修复，但没有主动阻塞影响 |
| **P4 — 不会修复/推迟** | 可接受的风险或超出当前产品范围 |

### Assignment

对于 `sprint` 模式下的每个 P1/P2 错误：
- 确定修复属于哪个故事或史诗
- 检查当前sprint是否有剩余容量
- 如果容量存在：分配给 sprint (`Sprint: [current]`)
- 如果容量已满：标记为`Priority overflow — consider pulling from sprint`

对于 `full` 模式：将所有 P1 分配给当前 sprint，将 P2 分配给下一个 sprint 估计，
P3+ to backlog.

### 偏差检查

标记表明**系统问题**的错误：
- 同一冲刺中同一系统的 3 个以上错误→“潜在的设计或
[system] 中的实现质量问题"
- 同一故事中存在 2 个以上 S1/S2 错误 →“故事可能需要重新打开并且
  发货前重新审核”
- 针对标记为“完成”的故事提交的错误→“已完成故事中的回归 -
  故事应该在冲刺跟踪中重新打开”

## 4. 趋势分析

对所有错误进行分类后，生成趋势指标：

### 成交量趋势
- 未解决的错误总数：[N]
- 开启本次冲刺：[N]
- 结束此冲刺：[N]
- 净变化：[+N / -N]

### 系统热点
- 哪个系统有最多的未解决的错误？
- 哪个系统的 S1/S2 比率最高？

### 年龄分析
- 有多少 bug 早于 2 个冲刺？
- 是否有任何 S1/S2 错误未分配（sprint = 无）？

### 回归指标
- 针对之前完成的故事提出了任何错误吗？
- 计数：[N] 回归错误（隐含故事重新开启）

## 6. Write 和门

在对话中展示报告，然后提问：

“我可以将这份分类报告写给 `production/qa/bug-triage-[date].md` 吗？”

Write 仅在批准后。

写完后：
- 如果任何 S1 bug 未分配：“S1 bug 必须在冲刺之前分配
  可以认为是健康的。运行 `/sprint-status` 查看当前容量。”
- 如果存在回归错误：“发现回归 - 考虑重新打开
  冲刺跟踪中受影响的故事并运行 `/smoke-check` 来重新选通。”
- 如果不存在 P1 错误：“没有 P1 错误 — 构建状况良好，适合 QA 移交。”结论：**完成** — 已撰写分诊报告。

如果用户拒绝写入：结论：**被阻止** - 用户拒绝写入。

---

## 协作协议

- **永远不要关闭或标记错误，未经用户批准就无法修复** - 将其暴露出来
  作为 P4 候选人并询问：“这些是否可以接受，因为不会修复？”
- **永远不要自动分配给容量最大的冲刺** - 标志溢出并让
  冲刺负责人决定要拉什么
- **严重性是客观的；优先级是团队决定** - 当前的严重性
  分类作为建议，而不是强制要求
- **趋势数据是信息性的**——不要仅仅阻碍趋势发现的工作；
  将它们作为观察结果呈现出来
