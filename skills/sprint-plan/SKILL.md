## 第 0 阶段：解析参数

提取模式参数（`new`、`update` 或 `status`）并解析审核模式（一次，存储本次运行的所有门生成）：
1. 如果 `--review [完整|lean|独奏]`已通过→使用
2. 否则请阅读 `production/review-mode.txt` → 使用该值
3. 否则 → 默认为 `lean`

有关完整检查模式，请参阅 `.codex/docs/director-gates.md`。

---

## 第一阶段：收集背景信息

1. **Read 是 `production/milestones/` 的当前里程碑**。

2. **Read 之前的冲刺**（如果有）从 `production/sprints/` 到
   了解速度和残留。

3. **扫描 `design/gdd/` 中的设计文档**以获取标记为就绪的功能
   以便实施。

4. **检查风险登记册** `production/risk-register/`。

## 第 2 阶段：生成输出

对于 `new`：

**按照此格式生成冲刺计划**并将其呈现给用户。不要要求写入 - 生产者可行性门（第 4 阶段）首先运行，并且可能需要在写入文件之前进行修改。

```markdown
# Sprint [N] -- [Start Date] to [End Date]

## Sprint Goal
[One sentence describing what this sprint achieves toward the milestone]

## Capacity
- Total days: [X]
- Buffer (20%): [Y days reserved for unplanned work]
- Available: [Z days]

## Tasks

### Must Have (Critical Path)
| ID | Task | Agent/Owner | Est. Days | Dependencies | Acceptance Criteria |
|----|------|-------------|-----------|-------------|-------------------|

### Should Have
| ID | Task | Agent/Owner | Est. Days | Dependencies | Acceptance Criteria |
|----|------|-------------|-----------|-------------|-------------------|

### Nice to Have
| ID | Task | Agent/Owner | Est. Days | Dependencies | Acceptance Criteria |
|----|------|-------------|-----------|-------------|-------------------|

## Carryover from Previous Sprint
| Task | Reason | New Estimate |
|------|--------|-------------|

## Risks
| Risk | Probability | Impact | Mitigation |
|------|------------|--------|------------|

## Dependencies on External Factors
- [List any external dependencies]

## Definition of Done for this Sprint
- [ ] All Must Have tasks completed
- [ ] All tasks pass acceptance criteria
- [ ] QA plan exists (`production/qa/qa-plan-sprint-[N].md`)
- [ ] All Logic/Integration stories have passing unit/integration tests
- [ ] Smoke check passed (`/smoke-check sprint`)
- [ ] QA sign-off report: APPROVED or APPROVED WITH CONDITIONS (`/team-qa sprint`)
- [ ] No S1 or S2 bugs in delivered features
- [ ] Design documents updated for any deviations
- [ ] Code reviewed and merged
```

对于 `status`：

**生成状态报告**：

## Progress: [X/Y tasks complete] ([Z%])

## 第 3 阶段：Write Sprint 状态文件

生成新的冲刺计划后，还要写入 `production/sprint-status.yaml`。
这是故事状态的机器可读真相来源 — 阅读者
`/sprint-status`、`/story-done` 和 `/help` 不带 Markdown 解析。

问：“我还可以写 `production/sprint-status.yaml` 来跟踪故事状态吗？”

Format:

```yaml
# Auto-generated by /sprint-plan. Updated by /story-done.
# DO NOT edit manually — use /story-done to update story status.

sprint: [N]
goal: "[sprint goal]"
start: "[YYYY-MM-DD]"
end: "[YYYY-MM-DD]"
generated: "[YYYY-MM-DD]"
updated: "[YYYY-MM-DD]"

stories:
  - id: "[epic-story, e.g. 1-1]"
    name: "[story name]"
    file: "[production/stories/path.md]"
    priority: must-have        # must-have | should-have | nice-to-have
    status: ready-for-dev      # backlog | ready-for-dev | in-progress | review | done | blocked
    owner: ""
    estimate_days: 0
    blocker: ""
    completed: ""
```

从冲刺计划的任务表中初始化每个故事：
- 必须完成的任务 → `priority: must-have`、`status: ready-for-dev`
- 应该有任务 → `priority: should-have`，`status: backlog`
- 很高兴有任务 → `priority: nice-to-have`，`status: backlog`

对于 `update`：读取现有的 `sprint-status.yaml`，结转状态
未更改的故事，添加新故事，删除删除的故事。

## 第四阶段：生产者可行性门

**审查模式检查** — 在生成 PR-SPRINT 之前应用：
- `solo` → 跳过。注意：“PR-SPRINT 已跳过 — 单人模式。”继续进行第 5 阶段（QA 计划门）。
- `lean` → 跳过（不是相位门）。注意：“PR-SPRINT 已跳过 — 精益模式。”继续进行第 5 阶段（QA 计划门）。
- `full` → 正常生成。

在最终确定冲刺计划之前，使用门 **PR-SPRINT** (`.codex/docs/director-gates.md`) 通过 Task 生成 `producer`。

通过：建议的故事列表（标题、估计、依赖项）、hours/days 中的团队总能力、之前冲刺的任何结转、里程碑约束和截止日期。

介绍制作人的评价。如果不现实，请在请求写入批准之前修改故事选择（将故事推迟到“应该有”或“最好有”）。如果有疑虑，请提出并让用户决定是否进行调整。

处理完制作人的判决后，问：“我可以把这个冲刺计划写到`production/sprints/sprint-[N].md`吗？”如果是，则写入文件，并根据需要创建目录。结论：**完成** — 冲刺计划已创建。如果否：判决：**被阻止** - 用户拒绝写入。

写入后添加：

> **范围检查：** 如果此冲刺包含超出原始史诗范围添加的故事，请在实施开始之前运行 `/scope-check [epic]` 以检测范围蔓延。

## 第五阶段：QA 计划门

在关闭冲刺计划之前，请检查此冲刺是否存在 QA 计划。

使用 `Glob` 查找 `production/qa/qa-plan-sprint-[N].md` 或 `production/qa/` 中引用此冲刺编号的任何文件。

**如果找到 QA 计划**：在 sprint 计划输出中记下它 - “QA 计划：`[path]`” - 并继续。

**如果不存在 QA 计划**：不要默默地继续。明确地表达这一点：

> “此冲刺没有 QA 计划。没有 QA 计划的冲刺计划意味着测试要求未定义 - 开发人员不会知道从 QA 的角度来看‘完成’是什么样的，并且如果没有计划，冲刺就无法通过生产 → 抛光门。
>
> 在开始任何实施之前，立即运行 `/qa-plan sprint`。它需要一个会话并生成每个故事所需的测试用例要求。”

使用 `AskUserQuestion`：
- 提示：“没有找到此冲刺的 QA 计划。您想如何继续？”
- Options:
  - `[A] Run /qa-plan sprint now — I'll do that before starting implementation (Recommended)`
  - `[B] Skip for now — I understand QA sign-off will be blocked at the Production → Polish gate`

如果 [A]：以“编写冲刺计划。接下来运行 `/qa-plan sprint` — 然后开始实施”结束。
如果[B]：在冲刺计划文档中添加警告块：

```markdown
> ⚠️ **No QA Plan**: This sprint was started without a QA plan. Run `/qa-plan sprint`
> before the last story is implemented. The Production → Polish gate requires a QA
> sign-off report, which requires a QA plan.
```

## 第六阶段：后续步骤

编写冲刺计划并解决 QA 计划状态后：

- `/qa-plan sprint` — **在实施开始之前需要** — 定义每个故事的测试用例，以便开发人员根据 QA 规范实施，而不是一张白纸
- `/story-readiness [story-file]` — 在开始之前验证故事是否已准备好
- `/dev-story [story-file]` — 开始实施第一个故事
- `/sprint-status` — 检查冲刺中期进度
- `/scope-check [epic]` — 在实施开始之前验证没有范围蔓延
